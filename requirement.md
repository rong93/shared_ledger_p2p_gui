# 分散式帳本 P2P 系統技術需求文件 (Requirement.md)

## 1. 需求背景
本專案旨在建構一個去中心化的「分散式帳本」原型系統。透過 Docker 容器模擬獨立的網路節點（礦工），利用 P2P 通訊協定同步交易紀錄，並導入密碼學雜湊鏈（Hash Chain）與分散式共識機制，確保帳本的完整性與一致性。

## 2. 技術實作細節

### 2.1 邏輯流程 (Flow)

1.  **交易發起與廣播**：
    *   使用者在任一節點輸入交易指令（例如：A 轉帳 10 元給 B）。
    *   發起節點透過 Socket 將交易訊息廣播至網路中所有已知節點的 IP。
    *   Socket 是用來實現 資料傳輸的手段。
2.  **同步寫入與檔案切分**：
    *   各節點接收到廣播訊息後，驗證格式並寫入本地當前帳本檔案。
    *   **滿額換檔機制**：當檔案內交易紀錄達到 5 筆時，系統自動封存該檔案並建立新檔案。
3.  **雜湊鏈連結 (Hashing)**：
    *   建立新檔案時，讀取前一檔案的全部內容並計算 SHA-256 雜湊值。
    *   將此雜湊值寫入新檔案的第一行，形成不可竄改的鏈狀結構。
4.  **共識驗證機制**：
    *   **本地驗證 (`checkChain`)**：逐一比對本地所有區塊檔案的雜湊值是否與記錄一致。
    *   **全球共識 (`checkAllChains`)**：節點向其他節點請求其最後一個區塊的雜湊值。若超過 2/3 的節點（3 節點中的 2 個）雜湊值相同，則視為帳本合法。
5.  **獎勵與餘額**：
    *   驗證通過後，發起驗證的節點獲得系統獎勵交易。
    *   系統透過掃描所有歷史檔案，加總特定使用者的所有交易流入流出以計算即時餘額。

## 3. 使用方法
系統啟動後，於容器終端機執行：
```bash
python3 p2p.py
```
進入互動式介面後，可輸入以下指令：
會顯示 Enter a command (checkMoney, checkLog, transaction, checkChain):
*   `transaction`：發起一筆新的交易。
*   `checkMoney`：查詢特定使用者的帳戶餘額。
*   `checkChain`：進行本地帳本雜湊鏈完整性檢查。
*   `checkLog`：啟動跨節點全球共識驗證。

## 4. 實作步驟清單

### 第一階段：基礎環境與資料結構
- [ ] **Docker 網路配置**：撰寫 `docker-compose.yml` 建立自定義網路並啟動 3 個 Ubuntu 容器。
- [ ] **檔案儲存邏輯**：實作每 5 筆交易自動建立新檔案（`ledger_N.txt`）的邏輯。
- [ ] **雜湊計算模組**：實作 SHA-256 函式，讀取前一檔案內容並產出雜湊值。
- [ ] **餘額查詢功能**：實作遍歷儲存目錄下所有檔案並計算特定 User 總額的函式。

### 第二階段：P2P 通訊開發
- [ ] **Socket Server**：開發背景監聽程式，負責接收 TCP/UDP 交易訊息。
- [ ] **廣播客戶端**：實作將本地發起的交易封包傳送至其餘節點 IP 的邏輯。
- [ ] **訊息持久化**：確保收到廣播訊息後，能正確寫入 `storage/` 目錄下的本地帳本。

### 第三階段：安全性與驗證機制
- [ ] **本地完整性檢查 (`app_checkChain.py`)**：實作掃描本地雜湊鏈是否斷裂的驗證工具。
- [ ] **跨節點共識驗證 (`app_checkLog.py`)**：實作 P2P 雜湊值比對邏輯與 2/3 多數決判定。
- [ ] **自動獎勵邏輯**：在 `checkAllChains` 通過後，自動產生一筆來源為 `SYSTEM` 的獎勵交易。

## 4. 潛在風險與注意事項

*   **網路競爭衝突 (Race Condition)**：若兩個節點同時發起交易，可能導致不同節點收到的交易順序不一。
    *   *對策*：初步實作可不考慮順序，但進階建議加入時間戳記 (Timestamp)。
*   **節點離線同步**：若某一容器暫時關閉後重啟，其本地帳本將與他人不同步。
    *   *對策*：啟動時應實作「追帳 (Sync)」機制，主動向鄰近節點請求缺失的檔案。
*   **效能瓶頸**：餘額計算需掃描所有歷史檔案，當檔案數量過多時會導致延遲。
    *   *對策*：可實作「狀態快照 (Snapshot)」或索引表加速查詢。
*   **Socket 阻塞**：同步寫入若採用阻塞式 Socket，可能因某一節點網路慢而卡死全網。
    *   *對策*：建議使用非阻塞式 IO 或多執行緒處理 Socket 通訊。
